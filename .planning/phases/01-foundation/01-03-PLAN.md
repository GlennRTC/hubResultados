Create `supabase/migrations/0000_rls_policies.sql`. This file serves as the canonical record of all Phase 1 RLS policies. It will be applied manually by the executor via the Supabase Dashboard SQL editor (or via `supabase db push` if Supabase CLI is available).

The RLS design follows this pattern:
- `laboratories`: a user can only access their own lab's row, found by joining to lab_users where auth_user_id = auth.uid()
- `lab_users`: a user can only see lab_users rows in their own lab

```sql
-- ============================================================
-- LabFlash RLS Policies — Phase 1
-- Apply in Supabase Dashboard: SQL Editor > Run
-- ============================================================

-- Enable RLS on all tables
ALTER TABLE laboratories ENABLE ROW LEVEL SECURITY;
ALTER TABLE lab_users ENABLE ROW LEVEL SECURITY;

-- ============================================================
-- laboratories policies
-- ============================================================

-- SELECT: lab staff can read their own lab's row
CREATE POLICY "lab_staff_select_own_lab"
  ON laboratories
  FOR SELECT
  USING (
    id IN (
      SELECT laboratory_id FROM lab_users WHERE auth_user_id = auth.uid()
    )
  );

-- INSERT: allow during registration (service role bypasses RLS anyway,
-- but anon/authenticated needs this for the server action that uses anon key)
-- Registration inserts are done via service role client — no INSERT policy needed for anon.

-- UPDATE: only admins can update their lab
CREATE POLICY "lab_admin_update_own_lab"
  ON laboratories
  FOR UPDATE
  USING (
    id IN (
      SELECT laboratory_id FROM lab_users
      WHERE auth_user_id = auth.uid() AND role = 'admin'
    )
  );

-- ============================================================
-- lab_users policies
-- ============================================================

-- SELECT: lab staff can see all users in their own lab
CREATE POLICY "lab_staff_select_own_lab_users"
  ON lab_users
  FOR SELECT
  USING (
    laboratory_id IN (
      SELECT laboratory_id FROM lab_users WHERE auth_user_id = auth.uid()
    )
  );

-- INSERT: only admins can add users to their lab (Phase 4 feature; safe to add now)
CREATE POLICY "lab_admin_insert_lab_users"
  ON lab_users
  FOR INSERT
  WITH CHECK (
    laboratory_id IN (
      SELECT laboratory_id FROM lab_users
      WHERE auth_user_id = auth.uid() AND role = 'admin'
    )
  );

-- DELETE: only admins can remove users from their lab
CREATE POLICY "lab_admin_delete_lab_users"
  ON lab_users
  FOR DELETE
  USING (
    laboratory_id IN (
      SELECT laboratory_id FROM lab_users
      WHERE auth_user_id = auth.uid() AND role = 'admin'
    )
  );
```

IMPORTANT: The registration server action (`registerAction` in 01-02) inserts into `laboratories` and `lab_users` using the anon-key Supabase client. This will fail with RLS enabled because the user's auth.uid() is set after signUp (the session may not be established yet within the same request).

To fix this, update `src/app/(auth)/register/actions.ts` to use the SERVICE ROLE key for the DB inserts:

In `src/lib/supabase/server.ts`, add a second exported function for admin operations:
```typescript
import { createClient as createSupabaseAdminClient } from "@supabase/supabase-js";

export function createAdminClient() {
  return createSupabaseAdminClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!,
    { auth: { autoRefreshToken: false, persistSession: false } }
  );
}
```

The Drizzle `db` client in `src/lib/db/index.ts` connects via DATABASE_URL (Supabase connection pooler). The pooler does NOT propagate Supabase auth JWT — so RLS policies on the `db` client will NOT see auth.uid(). This means:

- **`db` client (Drizzle)**: Use ONLY from server actions with service role behavior. For Phase 1 this is fine since we only do auth-gated server actions.
- **Supabase client**: Use for auth operations and client-side data fetching where RLS should apply.

Document this as a key architectural note in the SUMMARY.

Create `src/lib/auth/get-lab-user.ts` — a reusable server helper that verifies the session and loads the current user's lab context. Every dashboard Server Component calls this.

```typescript
import { redirect } from "next/navigation";
import { createClient } from "@/lib/supabase/server";
import { db } from "@/lib/db";
import { labUsers, laboratories } from "@/lib/db/schema";
import { eq } from "drizzle-orm";
import type { Laboratory, LabUser } from "@/lib/db/schema";

export type LabContext = {
  user: LabUser;
  lab: Laboratory;
};

export async function getLabUser(): Promise<LabContext> {
  const supabase = await createClient();
  const { data: { user }, error } = await supabase.auth.getUser();

  if (error || !user) {
    redirect("/login");
  }

  // Fetch lab_user row
  const labUser = await db.query.labUsers.findFirst({
    where: eq(labUsers.authUserId, user.id),
  });

  if (!labUser) {
    // Auth user exists but no lab_user — orphaned auth; sign out and redirect
    await supabase.auth.signOut();
    redirect("/login?error=cuenta_no_encontrada");
  }

  // Fetch laboratory
  const lab = await db.query.laboratories.findFirst({
    where: eq(laboratories.id, labUser.laboratoryId),
  });

  if (!lab) {
    redirect("/login?error=laboratorio_no_encontrado");
  }

  return { user: labUser, lab };
}
```

Note: `db.query.labUsers.findFirst` uses Drizzle's relational query API which requires the schema to be passed to drizzle() — already done in `src/lib/db/index.ts`. If for any reason it errors, use `db.select().from(labUsers).where(eq(labUsers.authUserId, user.id)).limit(1)` and destructure [0].
